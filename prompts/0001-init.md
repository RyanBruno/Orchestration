
## Goal

Build a new Claude Code orchestration system, including an HTML monitoring dashboard, from scratch, built on five principles: durable file-based state instead of anything living only in a running process or a model's context; mailbox-file messaging between workers instead of shared logs or direct calls; a single deterministic gather/process-inbox/advance/render tick loop per orchestrator, on a fixed interval; heartbeat-based liveness instead of trusting a worker is fine because nothing said otherwise; and a short, fixed, config-driven list of actions that always wait for a human, with everything else running unattended. The detailed goal list below is the actual spec.

## Goals this plan must achieve

1. **Durable state, not memory.** Every worker/task gets its own `state.json`: current phase, last action taken, what it's currently waiting on. It's overwritten on every change, never appended to. It's git-tracked or otherwise backed up, so a fresh process on the same or a different machine can reconstruct exactly where things stood by reading disk alone — it should never need to be told what happened by whatever's still in a model's context window. If a process is killed mid-task, its replacement must be able to pick up correctly from `state.json` alone.

2. **Mailbox-file messaging.** Build worker-to-worker communication as one JSON file per message, written into the recipient's `inbox/` folder, named so that timestamp ordering is unambiguous (e.g. an ISO-8601-ish prefix). The receiver processes strictly oldest-first and moves each file into `done/` after handling it — `done/` is a permanent record of who told whom what and when, not just cleanup. No two processes should ever append to the same file concurrently; separate files are what makes this safe without extra locking, since the filesystem's own atomicity does the work. A message to a sleeping or offline recipient should simply sit in `inbox/` until read — no timeout logic, nothing gets silently dropped.

3. **One deterministic tick loop per orchestrator.** Each orchestrator runs a fixed cycle (pick an interval — 5 minutes is a reasonable default — and document your reasoning): Gather (a fully deterministic script reads every worker's heartbeat and state files and computes what's eligible to run next — no model call here, no LLM judgment about whether to dispatch, only about what to dispatch once eligibility is code-determined), Process inbox (handle pending messages oldest-first, archive them), Advance (dispatch the next unit of work by writing to the target worker's inbox — but write the "I'm about to dispatch this" log entry *before* flipping any status flag, so a crash between the two leaves something a restart can unambiguously resolve instead of guessing whether it was already sent), and Render (produce a status/log update only when something actually changed — a quiet tick produces a quiet or zero-length log line, not repeated noise). Enforce a hard, code-level cap on how much can run "heavy" at once (e.g. only one large fan-out job system-wide at a time) — this must be a constant the code enforces, not a norm the model is expected to remember, since a model doesn't survive its own restart.

4. **Heartbeats instead of trust.** Every worker overwrites its own `heartbeat.json` at the end of each cycle: worker name, timestamp, an ok flag, its own currently-declared interval, and a one-line summary. A heartbeat older than roughly 2.5x the worker's own declared interval gets flagged stalled — generous enough that a worker legitimately running long (e.g. a slow analysis pass) isn't falsely flagged, provided it re-stamps its heartbeat with a realistic longer interval *before* starting the long task, not after. A worker being stopped on purpose must still write a heartbeat — a "stood down" state distinct from silence — so the dashboard reads intentional shutdown as idle, not as a crash. The dashboard should be able to answer "is anything broken?" from a single glance at last-heartbeat-vs-declared-interval per worker, not require investigation.

5. **A short, config-driven list of human gates.** Autonomy is the default; gates are the explicit, short exception list. Base it on these four categories: anything irreversible (deleting data, merging, sending something externally); anything crossing a trust boundary (a message leaving the system, a write to a repo/account not already explicitly authorized); anything ambiguous under current authorization (a worker that can't tell whether it's allowed to do something escalates rather than assuming broader permission for itself); and spend or scope/policy changes, as distinct from ordinary task execution. Implement this list as an easily-editable config (e.g. a `gates.yaml` or equivalent) rather than hardcoding it into logic, since the exact list is a risk-tolerance judgment call for the operator, not a purely technical one — propose a sensible default set based on the four categories above, but flag it clearly and prominently in the plan for the operator to review and adjust; don't treat your first draft as final or bury it. Items waiting on a human land in one place — an "operator-pending" queue or file — that the dashboard resurfaces every tick until disposed of; it must never silently expire or get buried in a log.

6. **Prove it works with example workers.** Since there's no existing production workload to test against, build at least two minimal example workers purely to exercise the coordination system end-to-end: one fast/near-instant worker, and one that simulates realistic long-running work (sleeps or does dummy work for a meaningful duration, long enough to exercise the heartbeat re-stamping behavior in goal 4). Keep these clearly marked as fixtures (e.g. under `examples/` or `fixtures/`), not production workers — they exist so every validation criterion below can actually be exercised, not left as an untested claim.
